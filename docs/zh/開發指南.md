> 社群維護的繁中導讀文件。內容為新增檔案，不修改上游原始檔，可安全合併上游更新。

## 常用指令

```bash
bun install              # 安裝相依套件
bun test                 # 跑免費測試（browse + 快照 + 技能驗證），<2 秒，每次 commit 前都跑
bun run dev <cmd>        # 開發模式跑 CLI，例如 bun run dev goto https://example.com
bun run build            # 生成文件 + 編譯 binary
bun run gen:skill-docs   # 從 .tmpl 模板重新生成所有 SKILL.md
bun run skill:check      # 所有技能的健康儀表板
bun run dev:skill        # watch 模式：改動時自動重生成 + 驗證
```

付費測試（需要 `ANTHROPIC_API_KEY`，發布前跑）：

```bash
bun run test:evals       # LLM judge + E2E，依 diff 自動選測試（單次上限約 $4）
bun run test:gate        # 只跑 gate 層測試（CI 預設，會擋合併）
bun run test:e2e         # 只跑 E2E（依 diff，單次上限約 $3.85）
bun run eval:select      # 預覽目前 diff 會觸發哪些測試
```

## 核心工作流：修改一個技能

技能文件是**生成的**，不能直接改最終檔。正確流程：

1. 編輯 `.tmpl` 模板（例如 `ship/SKILL.md.tmpl` 或 `SKILL.md.tmpl`）。
2. 若涉及共用邏輯，編輯對應的 `scripts/resolvers/*.ts`。
3. 跑 `bun run gen:skill-docs`（或 `bun run build`，會自動含這步）。
4. **同時 commit `.tmpl` 與生成的 `.md`**。

新增一個 `browse` 指令：加到 `browse/src/commands.ts` 再 rebuild。
新增一個 snapshot 旗標：加到 `browse/src/snapshot.ts` 的 `SNAPSHOT_FLAGS` 再 rebuild。

> **SKILL.md 的合併衝突永遠不要直接挑一邊。** 改去解 `.tmpl` 模板與 `gen-skill-docs.ts`（真正的來源），再跑 `bun run gen:skill-docs` 重新生成，最後 stage 重新生成的檔案。挑一邊會默默丟掉另一邊的模板改動。

## 寫 SKILL 模板的規則

`.tmpl` 是**給 Claude 讀的提示詞模板，不是 bash 腳本**。每個 bash code block 在獨立的 shell 執行，變數不會跨 block 保留。

- **用自然語言表達邏輯與狀態**，不要靠 shell 變數在 code block 之間傳遞狀態。
- **不要寫死分支名稱**，用 `gh pr view` / `gh repo view` 動態偵測 `main`/`master`。
- **每個 bash block 要能獨立運作**，需要前一步的 context 就在上方 prose 重述。
- **條件式用英文編號步驟表達**，不要在 bash 裡寫巢狀 `if/elif/else`。

## 平台無關設計（重要原則）

技能**絕不可**寫死框架專屬的指令、檔案模式或目錄結構。正確做法：

1. 讀 `CLAUDE.md` 拿專案專屬設定（測試指令、部署指令等）。
2. 沒有的話用 `AskUserQuestion` 問使用者。
3. 把答案寫回 `CLAUDE.md`，以後不用再問。

專案擁有自己的設定，gstack 只負責讀取。

## 測試分層

| 層級 | 內容 | 成本 |
|------|------|------|
| Tier 1 | 靜態技能驗證 + 生成器品質檢查 | 免費，<1 秒 |
| Tier 2 | E2E（透過 `claude -p`） | 付費，約 $3.85/次 |
| Tier 3 | LLM-as-judge 品質 eval | 約 $0.15/次 |

E2E 測試分 `gate` 與 `periodic` 兩層（定義在 `test/helpers/touchfiles.ts`）。CI 只跑 gate；periodic 每週 cron 或手動跑。新增 E2E 測試時要分類：安全護欄 / 決定性功能測試 → `gate`；品質基準 / Opus model / 非決定性 / 需外部服務 → `periodic`。

> **E2E 測試 fixture 不要整份複製 SKILL.md**（1500–2000 行會造成 context 爆炸與 timeout）。只抽出測試真正需要的段落。

## Commit 規範

- **永遠拆分可二分的 commit**：每個 commit 是單一邏輯改動。重新命名 / 移動與行為改動分開；測試基礎設施與測試實作分開；模板改動與生成檔重生分開。
- **暫存檔案永遠用明確檔名**（`git add file1 file2`），不要 `git add .` / `-A` — 會誤把編譯 binary 加進去。
- 只在使用者要求時 commit 或 push；在預設分支上要先開新分支。

## 開發時的 symlink 注意事項

開發 gstack 時，`.claude/skills/gstack` 可能是指回這個工作目錄的 symlink（gitignored）。這代表技能改動**立即生效**：

- 改 `.tmpl` + 跑 `gen:skill-docs` 會立刻影響所有 gstack 呼叫。
- 大重構時，半成品技能可能弄壞其他正在用 gstack 的 Claude Code session。
- 每個 session 開始先跑 `ls -la .claude/skills/gstack` 確認是 symlink 還是實體副本。大重構期間可移除 symlink（`rm .claude/skills/gstack`）改用全域安裝。

## 不要碰的東西

- **`browse/dist/`、`design/dist/`** 編譯 binary — 別 commit。
- **`ETHOS.md`** — Garry 個人的建造哲學，外部貢獻者不得修改。
- **促銷素材 / Garry 的語氣** — YC 引用、創辦人視角、產品聲音都是刻意的，不要「中立化」或「專業化」。

以上三類若要改動，需先透過 `AskUserQuestion` 取得使用者明確同意。
