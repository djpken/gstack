# gstack 技能總覽（繁體中文）

> 社群維護的繁中導讀文件。內容為新增檔案，不修改上游原始檔，可安全合併上游更新。
> 權威來源為各技能的 `SKILL.md` frontmatter 與 [`AGENTS.md`](../../AGENTS.md)；技能會增減，若有出入以英文版為準。
> 提示：你的輸出已設定為繁體中文，所有指令跟你互動時都會講中文。

gstack 的每個技能都是一個 slash command（斜線指令），代表一種專家角色。

這份文件分兩半：
- **上半／場景對照** — 「我在做什麼 → 該喊哪個指令」，照流程順序找。
- **下半／完整目錄** — 所有技能依角色分類，附逐一說明，當參考手冊查。

# 上半｜依場景選技能

## 每天核心迴圈（90% 的時間用這幾個）

```
想法 ──/office-hours──▶ 審查 ──/autoplan──▶ 實作 ──/review──▶ 測試 ──/qa──▶ 出貨 ──/ship──▶
```

| 我在做什麼 | 喊這個 | 它幫我 |
|-----------|--------|--------|
| 有個新功能/想法，想先想清楚 | `/office-hours` | 寫 code 前逼問值不值得做、最小切入點，產出設計文件 |
| 審查一張 ticket / issue / 計畫 | `/autoplan` | 一次跑 CEO+設計+工程+DX 四種視角審查 |
| 寫完一段 code，要審查 | `/review` | 找出能過 CI、卻會在 production 爆的 bug |
| 測網站功能、抓 bug | `/qa` | 開真瀏覽器跑功能、發現 bug 並修 |
| 要出 PR / 發版本 | `/ship` | 合併 base、跑測試、審 diff、bump 版本、寫 CHANGELOG、開 PR |

## 規劃階段（動工前）

| 場景 | 指令 |
|------|------|
| 需求很模糊，要變成精確可執行規格 | `/spec` |
| 只想要戰略把關：該不該做、範圍對不對 | `/plan-ceo-review` |
| 只想要工程把關：架構、邊界情況、測試 | `/plan-eng-review` |
| 只想要設計把關：UX、視覺層級 | `/plan-design-review` |
| 只想要開發者體驗把關：摩擦點、上手成本 | `/plan-devex-review` |
| 想從零打造一套設計系統 | `/design-consultation` |

> 偷懶版：以上規劃審查不確定挑哪個，直接 `/autoplan` 全跑。

## 實作與除錯

| 場景 | 指令 |
|------|------|
| 出現 bug / 錯誤，要找根因 | `/investigate` |
| 想要另一個 AI 的第二意見 | `/codex` |
| 怕 Claude 改到不該改的檔 | `/freeze`（限定目錄）→ 做完 `/unfreeze` |
| 要跑破壞性指令，想要安全護欄 | `/careful` 或 `/guard`（完整安全模式） |
| 想看專案整體程式碼品質 | `/health` |

## 常用組合

> 把你最常跑的流程記在這，下次直接照抄：

- **小修一個 bug**：`/investigate` → 改 → `/review` → `/ship`
- **做一個新功能**：`/office-hours` → `/autoplan` → 實作 → `/review` → `/qa` → `/ship`
- **審一張 ticket**：`/autoplan`（或單角度 `/plan-ceo-review` / `/plan-eng-review`）
- **（你自己的流程）**：_待補_

# 下半｜完整技能目錄

依角色分類，每條附用途說明，當參考手冊查。

## 1. 規劃與計畫審查（寫程式之前）

| 指令 | 用途 |
|------|------|
| `/office-hours` | **從這裡開始**。YC Office Hours 模式，在你寫程式前重新框定產品想法。產出設計文件，不寫程式。 |
| `/plan-ceo-review` | CEO / 創辦人視角的計畫審查 — 在需求裡找出「10 星產品」。 |
| `/plan-eng-review` | 工程主管視角 — 鎖定架構、資料流、邊界情況與測試。 |
| `/plan-design-review` | 設計師視角的計畫審查 — 各設計維度評 0–10 分並說明 10 分長怎樣。 |
| `/plan-devex-review` | 開發者體驗（DX）視角 — TTHW、魔法時刻、摩擦點、persona 追蹤。 |
| `/plan-tune` | 自我調校 AskUserQuestion 的提問敏感度與開發者心理輪廓。 |
| `/autoplan` | 一鍵跑完 CEO → 設計 → 工程 → DX 審查，用 6 條決策原則自動決定。 |
| `/design-consultation` | 從零打造完整設計系統（美學、字體、色彩、版面、間距、動態）。 |
| `/spec` | 五階段把模糊意圖變成精確可執行的 spec，開成 GitHub issue，可選擇 spawn 一個 agent。 |

---

## 2. 實作與程式碼審查

| 指令 | 用途 |
|------|------|
| `/review` | 上線前 PR 審查 — 找出能過 CI、卻會在 production 爆掉的 bug。 |
| `/investigate` | 系統化除錯，做根因調查。 |
| `/cso` | 資安長模式 — OWASP Top 10 + STRIDE 安全稽核。 |
| `/codex` | OpenAI Codex CLI 包裝，三種模式，提供跨 model 第二意見。 |
| `/health` | 程式碼品質儀表板。 |
| `/devex-review` | 即時開發者體驗稽核。 |
| `/learn` | 管理專案的 learnings（跨 session 累積的知識）。 |

---

## 3. QA 與瀏覽器自動化

| 指令 | 用途 |
|------|------|
| `/qa` | 系統化 QA 測試一個 web 應用，並修掉發現的 bug。 |
| `/qa-only` | 只回報、不修的 QA 測試。 |
| `/browse` | 快速 headless 瀏覽器，用於 QA 與 dogfooding。 |
| `/scrape` | 從網頁抓資料。 |
| `/skillify` | 把最近一次成功的 `/scrape` 流程固化成永久的 browser-skill。 |
| `/benchmark` | 用 browse daemon 偵測效能回歸。 |
| `/benchmark-models` | gstack 技能的跨 model 基準測試。 |
| `/setup-browser-cookies` | 從你真實的 Chromium 匯入 cookie 到 headless session。 |
| `/open-gstack-browser`（別名 `/connect-chrome`） | 啟動內建 sidebar 擴充的 AI 可控 Chromium。 |
| `/pair-agent` | 讓遠端 AI agent 與你的瀏覽器配對。 |

---

## 4. 視覺設計

| 指令 | 用途 |
|------|------|
| `/design-review` | 設計師之眼 QA — 找出視覺不一致、間距問題、層級問題、AI slop，然後修正。 |
| `/design-shotgun` | 生成多個 AI 設計變體，開比較看板，收集結構化回饋並迭代。 |
| `/design-html` | 設計定稿 — 生成 production 級的 Pretext-native HTML/CSS。 |

---

## 5. 發布與部署

| 指令 | 用途 |
|------|------|
| `/ship` | 發布流程：偵測並合併 base 分支、跑測試、審查 diff、bump VERSION、更新 CHANGELOG、commit、push、開 PR。 |
| `/land-and-deploy` | 落地並部署流程。 |
| `/canary` | 部署後的 canary 監控迴圈。 |
| `/landing-report` | workspace-aware ship 的唯讀佇列儀表板。 |
| `/setup-deploy` | 為 `/land-and-deploy` 設定部署參數（一次性）。 |
| `/document-release` | 發布後文件更新 + Diataxis 覆蓋率地圖。 |
| `/document-generate` | 從零為某功能 / 模組 / 整個專案生成缺漏的文件。 |

## 6. iOS 開發

| 指令 | 用途 |

## 8. 安全護欄與工作階段控制

## 怎麼選技能（快速路由）

- 產品想法 / 腦力激盪 → `/office-hours`
- 策略 / 範圍 → `/plan-ceo-review`
- 架構 → `/plan-eng-review`
- 設計系統 / 計畫審查 → `/design-consultation` 或 `/plan-design-review`
- 完整審查 pipeline → `/autoplan`
- bug / 錯誤 → `/investigate`
- QA / 測試網站行為 → `/qa` 或 `/qa-only`
- 程式碼審查 / 看 diff → `/review`
- 視覺打磨 → `/design-review`
- 發布 / 部署 / PR → `/ship` 或 `/land-and-deploy`
- 存進度 → `/context-save`；接回情境 → `/context-restore`
