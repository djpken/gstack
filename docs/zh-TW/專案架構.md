# gstack 專案架構與構成（繁體中文）

> 社群維護的繁中導讀文件。內容為新增檔案，不修改上游原始檔，可安全合併上游更新。
> 權威來源為英文 [`ARCHITECTURE.md`](../../ARCHITECTURE.md) 與 [`CLAUDE.md`](../../CLAUDE.md)。

---

## 一句話總覽

gstack **不是一個程式，而是三樣東西打包在同一個 repo 裡**：

1. **Skills（提示詞層）** — gstack 的靈魂，讓 Claude 扮演各種工程角色
2. **Binaries（程式碼層）** — 技能呼叫的實際工具（瀏覽器、設計、PDF）
3. **基礎設施** — 把前兩層黏起來的 CLI 工具、host 設定、建置腳本

理解這三層的差別，就理解了整個專案。

---

## 第 1 層：Skills（提示詞層）

54 個技能目錄（`ship/`、`office-hours/`、`review/`、`investigate/` …）。每個技能目錄結構：

```
ship/
├── SKILL.md.tmpl   ← 原始檔（編輯這個）
├── SKILL.md        ← 生成物（不要手動改）
└── sections/       ← 大型技能拆出的按需載入片段
```

### 關鍵觀念：SKILL.md 是「生成」出來的

Claude 實際讀的 `SKILL.md` 不是手寫的，而是由模板加上 resolver 生成：

```
SKILL.md.tmpl  +  scripts/resolvers/*.ts（45 個模組）
        │
        │   bun run gen:skill-docs   （scripts/gen-skill-docs.ts）
        ▼
   58 個 SKILL.md（最終 Claude 讀的提示詞）
```

- `.tmpl` 裡有 `{{PLACEHOLDER}}` 佔位符。
- `scripts/resolvers/` 負責把佔位符填成實際內容。共用邏輯（preamble、安全規範、寫作風格、review army）集中在這裡，再注入每個技能 — 所以有些 resolver 很大（`review.ts` 63KB、`design.ts` 58KB）。
- `model-overlays/`（`claude.md`、`gemini.md` …）是各 AI model 專屬的小段行為微調，生成時附加到結尾。

> **要改技能內容，永遠編輯 `.tmpl` 與 resolver，然後跑 `bun run gen:skill-docs` 重新生成，最後兩者一起 commit。** 直接改 `SKILL.md` 會在下次生成時被覆蓋。

## 第 2 層：Binaries（程式碼層）

約 645 個 TypeScript 檔，集中在三個會編譯成執行檔的子專案，外加一個 Chrome 擴充：

| 子專案 | 是什麼 | 規模 |
|--------|--------|------|
| **`browse/`** | 核心。Playwright headless 瀏覽器 CLI + server，供 QA / dogfooding 使用。含 sidebar 安全分層、PTY 終端、CDP 橋接 | 63 個 src + **146 個測試**，約 30MB |
| **`design/`** | 用 GPT Image API 生成設計變體 | 19 src + 9 測試 |
| **`make-pdf/`** | markdown → 出版級 PDF | 10 src + 8 測試 |
| **`extension/`** | Chrome 擴充（side panel + 終端機 + CSS inspector），純 JS | — |

`browse` 是真正的工程重心。CLAUDE.md 裡大半的架構警告（WebSocket 驗證、Unicode 清理、CDP 生命週期、tunnel 安全）都在講它。動到 `browse/src/server.ts`、`sidepanel.js` 等檔案前，先讀對應的設計文件。

> ⚠️ **`browse/dist/`、`design/dist/` 是編譯出來的執行檔，絕不要 commit。** 它們是 Mach-O arm64，只能在 Apple Silicon Mac 跑，`./setup` 會在每台機器自行編譯。`git status` 會顯示它們被改 — 忽略即可，且暫存檔案時永遠用明確檔名（`git add file1 file2`），不要 `git add .` / `-A`。

## 第 3 層：基礎設施（把前兩層黏起來）

| 部分 | 數量 | 作用 |
|------|------|------|
| **`bin/`** | 69 個 CLI 工具 | 技能的 bash preamble 呼叫的 `gstack-*` 小工具：`gstack-config`、`gstack-slug`、`gstack-brain-*`、`gstack-redact`、telemetry 等 |
| **`hosts/`** | 13 個設定檔 | 每個 AI agent 一個 typed 設定（`claude.ts`、`codex.ts`、`cursor.ts` …），決定技能如何對不同 host 生成 |
| **`scripts/`** | 76 個腳本 | 建置與開發體驗工具，`gen-skill-docs.ts` 是核心，`resolvers/` 也在此 |
| **`lib/`** | 10 個 | 跨專案共享（worktree、redact 引擎、env shim） |
| **`test/`** | 318 個 | 技能驗證 + LLM judge eval + E2E |
| **`setup`** | 1 個大型 bash | 一次性安裝：編譯 binary + 把技能 symlink 進 `~/.claude/skills/` |

## 三種檔案，三種命運（合併上游時最重要的一張表）

整個 repo 的檔案只分三類，改動它們的後果完全不同：

| 類型 | 例子 | 改它的後果 |
|------|------|-----------|
| **原始檔（source）** | `.tmpl`、`resolvers/*.ts`、`bin/*`、頂層 `.md` | 上游一改到同處就**衝突** |
| **生成物（generated）** | 58 個 `SKILL.md`、`llms.txt` | 手改 = 白工，`gen:skill-docs` 會覆蓋 |
| **編譯物（compiled）** | `*/dist/` 執行檔 | 別碰、別 commit |

> 如果你 fork gstack 並想長期跟上上游更新：**把客製化放在「新增的獨立檔案」**（例如這份繁中文件），就不會跟上游任何檔案衝突。修改原始檔越多，每次升級要解的衝突越多。

## 一個技能的生命週期（簡述）

1. 使用者輸入 `/office-hours` 之類的斜線指令。
2. Claude 讀取對應的 `SKILL.md`，從 **Preamble** 開始執行 — 跑 bash 抓 context（分支、設定、telemetry、brain 快取）。
3. 依技能的各 **Phase** 逐步進行（例如 office-hours：情境收集 → 提問 → 前提挑戰 → 替代方案 → 設計文件 → 收尾）。
4. 過程中可能呼叫 `bin/` 工具、`browse` binary、或其他技能。
5. 結束時寫入 **telemetry** 與 learnings。

要看某個技能到底會做什麼，直接讀它的 `SKILL.md`（生成後的完整版）即可。
